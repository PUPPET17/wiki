#大语言模型维基

使用大语言模型构建个人知识库的模式。

这是一个想法文件，它被设计成复制粘贴到你自己的大语言模型代理（例如OpenAI Codex、Claude Code、OpenCode/Pi等）。).它的目标是传达高层次的想法，但你的代理人将与你合作制定细节。

##核心理念

大多数人对大语言模型和文档的体验看起来像RAG：你上传一组文件，大语言模型在查询时检索相关块，并生成答案。这是可行的，但大语言模型正在从零开始重新发现每个问题的知识。没有积累。提出一个需要综合五份文件的微妙问题，大语言模型每次都要找到并拼凑相关片段。没有什么是建立起来的。NotebookLM、ChatGPT文件上传和大多数RAG系统都是这样工作的。

这里的想法不同。大语言模型**不仅仅是在查询时从原始文档中检索，而是逐步构建和维护一个持久的wiki**——一个位于您和原始源之间的结构化、相互链接的降价文件集合。当你添加一个新的来源时，大语言模型不仅仅是索引它以供以后检索。它阅读它，提取关键信息，并将其集成到现有的维基中——更新实体页面，修改主题摘要，注意新数据与旧主张相矛盾的地方，加强或挑战不断发展的综合。知识被编译一次，然后*保持最新*，而不是在每次查询时重新派生。

这是关键的区别：**维基是一个持久的、复合的工件。**交叉引用已经存在。矛盾已经被标记出来了。综合已经反映了你所读到的一切。随着你添加的每一个资源和你提出的每一个问题，维基变得越来越丰富。

你从不（或很少）自己编写维基——大语言模型编写并维护所有的维基。你负责采购、探索和提出正确的问题。大语言模型完成了所有繁重的工作——总结、交叉引用、归档和簿记，使知识库随着时间的推移变得有用。实际上，我让大语言模型特工在一边打开，黑曜石在另一边打开。大语言模型根据我们的对话进行编辑，我实时浏览结果——跟踪链接，检查图表视图，阅读更新的页面。黑曜石是IDE；大语言模型是程序员；维基是代码库。

这可以适用于许多不同的上下文。几个例子：

-**个人**：跟踪你自己的目标、健康、心理、自我提升——归档日记条目、文章、播客笔记，并随着时间的推移建立一个结构化的自我形象。
-**研究**：在几周或几个月内深入研究一个主题——阅读论文、文章、报告，并用不断发展的论文逐步构建一个全面的维基。
- **Reading a book**: filing each chapter as you go, building out pages for characters, themes, plot threads, and how they connect. By the end you have a rich companion wiki. Think of fan wikis like [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) — thousands of interlinked pages covering characters, places, events, languages, built by a community of volunteers over years. You could build something like that personally as you read, with the LLM doing all the cross-referencing and maintenance.
-**Business/team**：由大语言模型维护的内部wiki，由Slack threads、会议记录、项目文档、客户电话提供支持。可能有人在循环中查看更新。维基保持最新，因为大语言模型做了团队中没有人愿意做的维护工作。
-**竞争分析、尽职调查、旅行计划、课程笔记、爱好深入研究**-任何你随着时间的推移积累知识并希望它有组织而不是分散的东西。

##架构

有三层：

**原始来源**-您策划的源文件集合。文章、论文、图像、数据文件。这些是不可变的——大语言模型从它们中读取，但从不修改它们。这是你真相的来源。

**The wiki**-大语言模型生成的降价文件目录。摘要、实体页、概念页、比较、概述、综合。大语言模型完全拥有这一层。它创建页面，在新源到达时更新它们，维护交叉引用，并保持一切一致。你读了它；大语言模型写的。

**模式**-一个文档（例如CLAUDE.md代表Claude Code，AGENTS.md代表Codex），它告诉大语言模型维基是如何构建的，约定是什么，以及在获取资源、回答问题或维护维基时要遵循什么工作流程。这是关键的配置文件——它使大语言模型成为一个训练有素的维基维护者，而不是一个普通的聊天机器人。随着时间的推移，当你弄清楚什么适合你的领域时，你和大语言模型会共同发展这一点。

##操作

**摄取。**您将一个新源放入原始集合中，并告诉大语言模型处理它。一个示例流程：大语言模型读取源代码，与您讨论关键要点，在wiki中编写摘要页面，更新索引，更新整个wiki中相关的实体和概念页面，并在日志中附加一个条目。一个来源可能会接触10-15个维基页面。就我个人而言，我更喜欢一次获取一个来源并保持参与——我阅读摘要，检查更新，并指导大语言模型强调什么。但是您也可以在较少监督的情况下一次批量摄取许多源。由您来开发适合您风格的工作流，并将其记录在模式中以供将来的会话使用。

**查询。**您针对维基提出问题。大语言模型搜索相关页面，阅读它们，并用引用综合一个答案。根据问题的不同，答案可以采取不同的形式——降价页面、对照表、幻灯片（Marp）、图表（matplotlib）、画布。重要的见解：**好的答案可以作为新页面归档回wiki。**你要求的比较、分析、你发现的联系——这些都是有价值的，不应该消失在聊天记录中。这样，你的探索就像摄入的资源一样在知识库中复合。

**棉绒。**定期要求大语言模型对维基进行健康检查。寻找：页面之间的矛盾，新来源已经取代的陈旧声明，没有入站链接的孤立页面，提到了重要概念但没有自己的页面，缺少交叉引用，可以通过网络搜索填补的数据空白。大语言模型善于提出要调查的新问题和寻找的新来源。这使得维基在成长过程中保持健康。

##索引和日志

随着维基的发展，两个特殊的文件帮助大语言模型（和你）浏览它。它们有不同的用途：

**index.md**是面向内容的。它是wiki中所有内容的目录——每个页面都列出了一个链接、一行摘要，以及可选的元数据，如日期或来源计数。按类别（实体、概念、来源等）组织。）.大语言模型每次摄入都会更新。在回答查询时，大语言模型首先读取索引以找到相关页面，然后深入其中。这在中等规模（大约100个源代码，大约数百页）下工作得非常好，并且避免了对基于嵌入的RAG基础设施的需求。

**log.md**按时间顺序排列。这是一个仅附加的记录，记录了发生了什么以及何时发生的情况——摄取、查询、lint传递。一个有用的提示：如果每个条目都以一致的前缀开头（例如`##【2026-04-02】ingest|Article Title`），日志就可以用简单的unix工具解析——`grep”^##\【”log.md|tail-5`给出最后5个条目。该日志给你一个维基发展的时间表，并帮助大语言模型了解最近做了什么。

##可选：CLI工具

At some point you may want to build small tools that help the LLM operate on the wiki more efficiently. A search engine over the wiki pages is the most obvious one — at small scale the index file is enough, but as the wiki grows you want proper search. [qmd](https://github.com/tobi/qmd) is a good option: it's a local search engine for markdown files with hybrid BM25/vector search and LLM re-ranking, all on-device. It has both a CLI (so the LLM can shell out to it) and an MCP server (so the LLM can use it as a native tool). You could also build something simpler yourself — the LLM can help you vibe-code a naive search script as the need arises.

##提示和技巧

-**Obsidian Web Clipper**是一个浏览器扩展，可将Web文章转换为markdown。对于快速将资源放入您的原始收藏中非常有用。
-**将图像下载到本地。**在黑曜石设置→文件和链接中，将“附件文件夹路径”设置为固定目录（例如`raw/assets/`）。然后在设置→热键中，搜索“下载”找到“下载当前文件的附件”并将其绑定到热键（例如Ctrl+Shift+D）。剪辑完一篇文章后，点击热键，所有的图片都会下载到本地磁盘。这是可选的，但很有用——它让大语言模型直接查看和引用图像，而不是依赖可能会中断的URL。请注意，大语言模型无法一次原生阅读带有内联图像的markdown——解决方法是让大语言模型先阅读文本，然后单独查看部分或全部引用的图像，以获得额外的上下文。它有点笨重，但工作得很好。
-**Obsidian的图形视图**是查看wiki形状的最佳方式——什么连接到什么，哪些页面是中心，哪些是孤立的。
-**Marp**是基于降价的幻灯片格式。黑曜石有一个插件。可用于直接从wiki内容生成演示文稿。
-**Dataview**是一个Obsidian插件，可以在页面frontmatter上运行查询。如果您的大语言模型将YAML frontmatter添加到wiki页面（标签、日期、来源计数），Dataview可以生成动态表格和列表。
-wiki只是markdown文件的git repo。您可以免费获得版本历史、分支和协作。

##为什么这有效

维护知识库的乏味部分不是阅读或思考，而是记账。更新交叉引用，保持摘要最新，注意新数据何时与旧声明相矛盾，保持几十页之间的一致性。人类放弃维基是因为维护负担的增长快于价值的增长。大语言模型不无聊，不忘更新一个交叉引用，一遍就能摸15个文件。维基保持维护，因为维护成本接近于零。

人类的工作是整理资料来源，指导分析，提出好的问题，并思考这一切意味着什么。大语言模型的工作就是其他一切。

这个想法在精神上与Vannevar Bush的Memex（1945）有关——一个个人的、策划的知识商店，在文档之间有关联路径。布什的愿景更接近这一点，而不是网络变成了什么样子：私人的，积极策划的，文件之间的联系和文件本身一样有价值。他解决不了的部分是谁来维护。大语言模型会处理的。

这份文件有意抽象。它描述了想法，而不是具体的实现。确切的目录结构、模式约定、页面格式、工具——所有这些都将取决于您的域、偏好和您选择的大语言模型。上面提到的一切都是可选的和模块化的——选择有用的，忽略没用的。例如：您的源可能是纯文本的，因此您根本不需要图像处理。你的wiki可能足够小，索引文件就是你所需要的，不需要搜索引擎。你可能不关心幻灯片，只想要降价页面。您可能需要一组完全不同的输出格式。正确的使用方法是与您的大语言模型代理共享它，并一起实例化一个符合您需求的版本。文档的唯一工作是传达模式。你的大语言模型可以解决剩下的问题。
